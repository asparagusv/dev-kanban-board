# Dev Kanban Board - 開発ジャーニーログ

**Dev Kanban Board** の開発ジャーニーへようこそ。このプロジェクトは、モダンな React 開発、厳格な TypeScript、そして開発者目線のデザインを追求したショーケースです。

## 🚀 ビジョン
このプロジェクトの目的は、以下の点に焦点を当てた、堅牢で高パフォーマンスな開発者向けカンバンボードを構築することでした：
- **パフォーマンス**: Vite と React 18 を活用し、高速な HMR を実現。
- **型安全 (Type Safety)**: 厳格な TypeScript 設定を使用し、コードの信頼性を担保。
- **UX**: 直感的なドラッグ＆ドロップ、永続的なステート、洗練されたダークモードの実装。

## 🛠 使用技術
- **フレームワーク**: [React 18](https://react.dev/)
- **ビルドツール**: [Vite](https://vitejs.dev/)
- **言語**: [TypeScript](https://www.typescriptlang.org/)
- **アイコン**: [Lucide-React](https://lucide.dev/)
- **アーキテクチャ**: カスタムフック、および `React.lazy` と `Suspense` によるコード分割。

### 💡 JSON.parse() の `any` 対策

LocalStorage からデータを持ってくるとき、`JSON.parse()` を使うけど...
これ、デフォルトだと **`any` 型** が返ってくる。

👉 つまり、**TypeScript の守護が消える瞬間**。

#### どう対策する？
ここで型アサーション (`as Task[]`) の出番。
```typescript
return JSON.parse(savedTasks) as Task[];
```
これで、「このデータは `Task` の配列だよ！」と明示的に宣言する。

#### これで勝てる理由
-  ストレージのデータと UI コンポーネントの間に **「契約」** が結ばれる。
-  万が一データが壊れていても、型エラーで早期に気づける。
-  実行時の「謎のエラー」を未然に防ぎ、開発体験を爆上げ。

### 💡 親子がバラバラにならないための「ステートのリフティング」

コンポーネントを分けると、誰が「正解（ステート）」を持てばいいか迷子になりがち。
今回採用したのが **「State Lifting up」** という鉄板パターン。

#### 1. 子（TaskCard）は「叫ぶ」だけ
子コンポーネントは、タスクを動かす権利を持たない。
ボタンが押されたら、Props で受け取った関数を呼んで「右に動かしたい！」と親に叫ぶだけ。
```tsx
<button className="move-button" onClick={() => onMoveTask(task.id, 'right')}>
  次へ →
</button>
```

#### 2. 親（App.tsx）が「責任」を取る
親は `tasks` ステートの管理者。
子の叫びを聞いて、イミュータブルに（元の配列を壊さずに）新しい状態を作る：
```typescript
const handleMoveTask = (id: number, direction: 'left' | 'right') => {
  setTasks((prevTasks) => 
    prevTasks.map((task) => {
      if (task.id !== id) return task;
      // 順序（statusOrder）に基づいて安全にステータスを更新
      return { ...task, status: statusOrder[nextIndex] };
    })
  );
};
```

#### なぜこれが最強？
-   **単一責任の原則**: 状態変更のロジックが親に集まるから、デバッグが楽。
-   **データの一貫性**: あちこちでステートをいじらないから、不整合が起きない。

### 💡 なぜ statusOrder を App.tsx に置かないのか？

まず前提の整理。`statusOrder` はこれ👇
```typescript
export const statusOrder: TaskStatus[] = ['todo', 'in-progress', 'done'];
```
これってただの配列じゃなくて、**「ステータスの遷移ルール」**そのもの。

#### なぜ App.tsx に置かない？
App.tsx に置いちゃうと...
-  ロジックが UI と混ざる
-  他のコンポーネントから使えない（再利用できない）
-  結果、あちこちに同じ定義が散らばる

#### 判断の基準
迷ったときはこれ👇
> **それは「UI」か？ それとも「ドメイン（ルール）」か？**

`statusOrder` の正体は：
-  UI じゃない
-  データのルール
-  状態の順番定義

👉 つまり **「ドメインロジック」**。だから `types/index.ts` に置くべき。

#### types/index.ts に集約した理由
-  **TaskStatus と密接に関係してる**: 型定義とそのルール（順序）は同じ概念のグループ。
-  **1箇所で管理**: 「ステータスの定義 + 順序」を同じ場所に集約。

もし分けなかったら？
あちこちで `const statusOrder = [...]` を定義することになり、将来的に **「値がズレる」「バグる」「修正地獄」** を見ることになります。

#### 💡 設計の勝てるルール
**「型とそのルールは近くに置く」**
-   `TaskStatus`（型）
-   `statusOrder`（ルール）
👉 これらは同じファイルに置く。これだけでコードの堅牢性が一気に上がります。

### 🚀 カスタムフックとジェネリクスの力

`App.tsx` に直接書いていた LocalStorage のロジックを、`useLocalStorage` というカスタムフックに抽出しました。ここで輝くのが **「ジェネリクス (`<T>`)」** です。

#### なんでジェレニクスを使う？
LocalStorage は文字列しか保存できないけど、アプリ側では `Task[]` や `UserConfig` などの複雑なデータを扱いたい。
```typescript
export function useLocalStorage<T>(key: string, initialValue: T) {
  // ...
}
```
この `<T>` は **「使うときに型を決める」** という予約席のようなもの。

#### 呼び出し側（App.tsx）
```typescript
const [tasks, setTasks] = useLocalStorage<Task[]>('kanban-tasks', initialTasks);
```
👉 `T` に `Task[]` を入れることで：
1.  `useState` の中身が自動的に `Task[]` になる。
2.  `setTasks` が `Task[]` 以外の入力を弾く。
3.  保存するときの `JSON.stringify` も安全に行える。

#### メリット
- **再利用性**: タスク以外のデータ（例：設定、ユーザー情報）にもそのまま使い回せる。
- **抽象化**: `App.tsx` が「ストレージの保存方法」を知らなくて済む。
- **堅牢性**: `as T` でパース結果に型を付けることで、フック内部でも型安全をキープ。

### 🚀 useTasks カスタムフックへの集約と「関心の分離」

以前は `App.tsx` に書かれていた膨大なタスク操作ロジックを、`useTasks` という独自のカスタムフックへと切り出しました。

#### なぜ切り出したのか？（判断の基準）

1. **同じ「関心事」に対する操作の集約**
   タスクの「追加、削除、更新、矢印移動、ドラッグ移動、WIP制限の判定、フィルタリング（memo化）」... これらはすべて **`tasks` という一つのステートに対して行われる操作** です。
   一箇所にまとめることで、「タスクをどう扱うか」というルール（ドメインロジック）の凝集度が高まり、変更にも強くなりました。

2. **「UI」と「ロジック」の徹底した分離**
   `App.tsx` は本来、どのコンポーネントをどこに並べ、どの Props を渡すかという **「見た目の宣言（What）」** に集中すべき場所です。そこに具体的な操作手順（How）が混ざると、コードを読む際のノイズになります。
   ロジックをフックに隠蔽することで、`App.tsx` を読み返したときに「あ、ここはカンバンボードの画面なんだな」という意図が瞬時に伝わるようになりました。

#### 切り出し後のメリット
- **App.tsx が劇的にスッキリ**: わずか 80行程度になり、全体の構造が俯瞰しやすくなりました。
- **メンテナンス性の向上**: タスク操作にバグがあったり、仕様を変更したいときは `useTasks.ts` だけを見ればよくなりました。
- **副作用の局所化**: `App.tsx` 全体を再レンダリングさせることなく、特定のロジックだけを最適化（memo化など）しやすくなります。

> **「画面のコードを読みたいときに、ロジックが邪魔にならない」。**

これが、読みやすくメンテナンスしやすい React アプリケーションを作るための鉄則です。

### 💡 カスタムフックの中身、どう動いてる？

`useLocalStorage` の中身、実は毎レンダリングごとに上から順に実行されています。「カスタムフックは単にフックをまとめた**普通の関数**」だからです。

#### 1. 初回レンダリングの動き
1.  **`useLocalStorage` が呼ばれる**: 関数が上から実行開始。
2.  **`useState` の初期化**: React が「初回だ！」と判断し、初期化関数 `() => { ... }` を実行。ここで `localStorage.getItem` が走る。
3.  **`useEffect` の登録**: 「レンダーが終わったらこれ動かしてね」と React に予約を入れる。
4.  **値を返す**: `[value, setValue]` を返して `App.tsx` の JSX が作られる。
5.  **コミット後**: 画面に反映された **後** に、予約していた `useEffect` が実行される。

#### 2. 2回目以降（再レンダリング）
1.  **再度上から実行**: `useLocalStorage` はまた最初から実行されます。
2.  **useState は毎回呼ばれる？**: 呼ばれます。
でも React は「もう state 持ってるよ」と知っているので、初期化関数（`useState(() => { ... })` の中身）は**スルーされます**。

    ```typescript
    const [value, setValue] = useState<T>(() => {
      console.log('initializer'); // これは最初しか出ない！
      return ...;
    });
    ```
3.  **useEffect のチェック**: 依存配列 (`[key, value]`) を見て、変更があればコミット後に中身を実行する。

👉 **ここがポイント！**
> **「フックは飛ばされている」のではなく、「毎回呼ばれているが、初回と2回目以降で振る舞いが違う」**

これが React Hooks の魔法の正体。カスタムフックを作ることで、この複雑なライフサイクル管理を `App.tsx` から隠蔽できるのが最大の強みです。

#### 💡 [Update] 型ガードの融合
`useLocalStorage` に第3引数として型ガード (`isValid`) を渡せるようにしました。
```typescript
const [tasks, setTasks] = useLocalStorage('tasks', initial, isTaskArray);
```
これにより、LocalStorage に「もし壊れたデータ」が入っていても、初期化の段階で自動的に検知し、安全に初期値へ戻すことができます。

### 💡 カスタムフックの戻り値と「as const」の魔法

`useLocalStorage` の最後で `[value, setValue] as const` と書いているのには、深い理由があります。

#### 配列 (Array) vs タプル (Tuple)
TypeScript にとって、普通の配列とタプルは別物です：
- **配列**: 「長さも中身も不定」。何個入るかわからない。` (T | Dispatch)[]` と推論される。
- **タプル**: 「順番と型が固定」。決まったセット。`[T, Dispatch]` と推論される。

#### なぜ `as const` が必要なのか？
もし `as const` を付けずに `return [value, setValue]` と書くと、TypeScript はデフォルトで「これは要素が入れ替わるかもしれないただの配列だ」と判断し、`(T | Dispatch<...>)[]` という広すぎる型をつけてしまいます。これでは使う側で値と関数を正しく区別できません。

#### `as const` を付けると...
```typescript
return [value, setValue] as const;
```
👉 TypeScript に **「これは 2要素の固定セット（タプル）だよ！」** と定義を押し付けられます。その結果、型は `readonly [T, Dispatch<...>]` となり、`useState` と全く同じ感覚で安全に分割代入して使えるようになります。

### 🚀 useMemo で「無駄な計算」をバッサリ削る

カンバンボードでは、全タスクの中から「Todo」「進行中」「完了」を仕分けして表示します。ここで登場するのが **`useMemo`** です。

#### なんで `useMemo` が必要なのか？
もし `useMemo` を使わずにこう書くと...
```typescript
const todoTasks = tasks.filter((task) => task.status === 'todo');
const inProgressTasks = tasks.filter((task) => task.status === 'in-progress');
const doneTasks = tasks.filter((task) => task.status === 'done');
```
👉 **毎レンダリングごとに 3回 フィルターが走ります。**

たとえばタスクが 100個あれば、レンダリングのたびに 300回のチェックが実行される計算。「入力フォームに 1文字打つ（ステート更新）」だけでも、この重い計算が毎回走ってしまいます。

#### 賢い解決策
```typescript
const todoTasks = useMemo(() => tasks.filter((task) => task.status === 'todo'), [tasks]);
```
👉 これで React に **「`tasks` が変わったときだけ計算して、それ以外は前回の結果を使い回して！」** と命令できます。

#### メリットまとめ
- **パフォーマンス向上**: 無駄なループ処理を最小限に抑えられる。
- **カクつき防止**: タスク量が増えても、入力フォームなどの操作がサクサク動く。
- **依存関係の明確化**: 「このデータは `tasks` に依存している」というコードの意図がハッキリする。

> **「重い計算はメモ化する」。これだけで、アプリの「軽さ」が変わります。**

### 🚀 React.lazy で「必要なときだけ」読み込む

`useMemo` が「計算の効率化」なら、**`React.lazy`** は「ファイル読み込みの効率化」です。`Column` コンポーネントを遅延読み込みするように変更しました。

#### lazy を使う本当の理由
「処理が重いから `lazy` にする」というのは、実は少しズレています。

1.  **初期ロードを爆速にする** ✅
    最初に読み込む JS ファイル（メインの塊）を小さくできます。ユーザーがサイトを開いたときの「操作可能になるまでの時間 (TTI)」を短縮するのが狙いです。
2.  **「サイズが大きい」コンポーネントを分ける** ✅
    処理が重いかどうかではなく、**「ファイルサイズが大きい（依存ライブラリが多い）」** コンポーネントが対象。チャート系（chart.js）やエディタ（Monaco）などをイメージすると分かりやすいです。
3.  **「条件分岐」で現れるものに使う** ✅（これが本質！）
    `{isOpen && <Modal />}` のように、最初は隠れているコンポーネント。これ、必要になるまでダウンロードすらしないのが正解です。

#### 今回の実装
```typescript
const Column = lazy(() => import('./components/Column'));

// ...

<Suspense fallback={<div>Loading...</div>}>
  <div className="board">
    <Column ... />
  </div>
</Suspense>
```
👉 `Column` コンポーネントが実際に画面に必要になったタイミングで、React が裏側でファイルを読みに行きます。

#### まとめ
> **「JS を一度に全部読み込まない」。**

ユーザーに必要な分だけを届けることで、最初の「お待たせする時間」を最小限に抑えるのが `lazy` の本質です。

### 🚀 HTML5 ドラッグ＆ドロップで「直感的な操作」を

ボタン操作だけじゃない。カンバンボードなら当然欲しい **「ドラッグ＆ドロップ」** を、外部ライブラリを使わず HTML5 標準 API だけで実装しました。

#### 3段階のシンプルな流れ

1.  **[出発] `TaskCard` で ID をメモする**
    `draggable` を有効にして、ドラッグ開始時に `taskId` を `dataTransfer` に保存します。
    ```tsx
    <div draggable onDragStart={(e) => e.dataTransfer.setData('taskId', task.id.toString())}>
    ```

2.  **[空中] `Column` で受け入れ態勢を作る**
    デフォルトだとドロップは拒否されるので、`onDragOver` で `e.preventDefault()` して「ここは落としてもいいよ！」と宣言します。

3.  **[到着] `Column` でステートを書き換える**
    ドロップされた瞬間、メモしていた `taskId` を取り出し、その列の `targetStatus`（Todo, In Progress など）へタスクをワープさせます！
    ```typescript
    const handleMoveTaskDrag = (taskId: number, targetStatus: TaskStatus) => {
      setTasks((prev) => 
        prev.map((task) => (task.id === taskId ? { ...task, status: targetStatus } : task))
      );
    };
    ```

#### 💡 深掘り：onDrop はどう動く？
ここ、意外とハマりやすいポイントです。

-   **どこで発火する？**: `onDrop` は「マウスが離された場所」にある要素で実行されます。何がドラッグされているかは関係ありません。**「どこに落としたか」** だけで決まります。
-   **必須条件**: `onDragOver` で `e.preventDefault()` を呼んでいないと、ブラウザのデフォルト挙動に邪魔されて `onDrop` は**絶対に発火しません**。
-   **データの橋渡し**: `onDrop` 自体は「何が落ちてきたか」を知りません。だからこそ、Step 1 で `dataTransfer.setData` し、Step 3 で `dataTransfer.getData` することで、コンポーネント間をまたいで ID を受け渡しているのです。

#### なんでこれでいいの？
- **ライブラリ不要**: `react-beautiful-dnd` などを使わなくても、標準機能だけで十分に動きます。
- **軽量**: 余計な JS を読み込まないので、アプリが軽いまま。
- **標準の力**: ブラウザ標準の機能を使うことで、信頼性の高い操作感が得られます。

### 💡 デザインとデータ（UI とドメイン）を混ぜない

以前は `Column` 内で表示用の `title` を無理やりステータスに変換していました：
```typescript
onDrop={(e) => handleDrop(e, title.toLowerCase() as TaskStatus)}
```
👉 **これ、実はバグの温床です。**

#### なんで「title から変換」がダメなのか？
「Todo」は `todo` で動きますが、「In Progress」は `in progress` になってしまい、定義した `in-progress`（ハイフンあり）とズレてしまいます。
わざわざ `titleToStatus` みたいな「意味不明な変換マップ」を作る羽目になります。

- **UI（見た目）**: ユーザーには「In Progress」と見せたい。
- **ドメイン（論理）**: システム内部では `in-progress` という型で扱いたい。

#### 賢い解決策：Props を分ける
表示用の `title` とは別に、論理的な `status` を Props として明示的に受け取るように変更しました。
```tsx
<Column title="In Progress" status="in-progress" ... />
```

#### メリット
- **型安全**: `as TaskStatus` みたいな「型のごまかし」が不要になります。
- **疎結合**: 表示名を「進行中」に変えても、システム側のロジック（`status`）に影響が出ません。
- **シンプル**: 無駄な文字列変換ロジックが消え、コードがスッキリします。

> **「表示用の名前」と「システム用の値」は、最初から分けて考える。**

これが、変化に強いコードを書くための鉄則です。

### 🚀 WIP 制限で「マルチタスク」を防ぐ

カンバンの真髄は **WIP (Work In Progress) 制限** にあります。仕掛中のタスクが増えすぎないよう、システムで制限をかけました。

#### TypeScript の工夫：Partial<Record> の使い道
制限を定義する際、`Record` ではなく **`Partial<Record>`** を使っています。

- **`Record<TaskStatus, number>` ❌**
  「すべてのステータスに数字を入れろ！」という命令。`todo` も `done` も、全部に制限を書かないとコンパイルエラーになります。
- **`Partial<Record<TaskStatus, number>>` ✅**
  「必要なステータスだけ制限していいよ」という柔軟な設定。今回は `in-progress` だけを 3つに制限し、ほかは無制限にしています。

#### 実装のこだわり
- **ガードコード**: タスクを移動する際、移動先のカウントをチェック。制限を超えていれば移動（ワープ）を拒否します。
- **視覚的なフィードバック**: ドラッグ中に列をまたぐと、点線の枠線（`column-drag-over`）が出るように `isDragOver` ステートを追加。また、ヘッダーに現在のタスク数と制限を表示(`(2/3)` など)し、上限に達した際は警告を表示するようにしました。

#### まとめ
> **「全部やらなくていい。今やることを絞る」。**

技術的な制約（TypeScript）と、実務的な制約（WIP）を組み合わせることで、道具としてのカンバンボードを一段上のレベルに引き上げました。

### 🚀 ユーザー定義型ガードで「不純物」を徹底排除

外部からの JSON インポート機能を実装しました。ここで、`any` や `unknown` という不確実な世界を、安全な TypeScript の世界へと変換する **「ユーザー定義型ガード」** の出番です。

#### なぜ `as Task[]` ではダメなのか？
以前は `as Task[]` という型アサーションを使っていました。これは「俺を信じろ、これは絶対 Task[] だ！」と TypeScript に**嘘をつく（上書きする）**行為です。もし JSON の中身が壊れていても、TypeScript はそれをスルーしてしまい、実行時に突然アプリが落ちる原因になります。

#### 魔法の構文：`data is Task[]`
そこで導入したのが、戻り値の型に **`data is Task[]`** を指定した検証関数です。

```typescript
export function isTaskArray(data: unknown): data is Task[] {
  // ... 全ての項目をチェックして OK なら true を返す
}
```

この `is` 構文は、単なる `boolean` 以上の意味を持ちます。「この関数が `true` を返したなら、**そのスコープ内では `data` を `Task[]` として扱っていいよ！**」と TypeScript のコンパイラに論理的な証明を提示しているのです。

#### 堅牢なインポートフロー
```typescript
const parsed = JSON.parse(text); // ここでは any (型が不明)

if (isTaskArray(parsed)) {
  // ここに入った瞬間、parsed は「any」から「Task[]」に劇的に変化（型絞り込み）！
  setTasks(parsed); // 型安全にセットできる
} else {
  alert('データの形式が正しくありません');
}
```
👉 **「開発者の思い込み (`as`)」** を **「実行時の確証 (`is`)」** に置き換える。これが、外部データと共存するための最も安全で、かつ TypeScript らしいやり方です。

#### まとめ
> **「型を信じるな、定義を疑え」。**

`data is Task[]` という誓約によって、不確かな外部データを「信頼できるデータ」へと昇華させることができました。
